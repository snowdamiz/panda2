# Advanced Discord LLM Assistant Bot Plan

## Reader And Outcome

This plan is for the engineer who will build the first production-ready version of the bot. After reading it, they should be able to scaffold the Go service, implement the core assistant and admin features, and know what to defer until later milestones.

## Decision Summary

- Backend language: Go.
- Discord integration: use `github.com/disgoorg/disgo` as the primary Discord library because it has higher-level coverage for application commands, components, modals, interactions, gateway events, sharding, and REST calls. Keep the Discord adapter behind an internal interface so the project can swap to `github.com/bwmarrin/discordgo` if DisGo creates friction.
- HTTP framework: use Go Fiber for health checks, metrics endpoints, internal admin endpoints, Discord interaction webhooks if needed later, and any future web dashboard API.
- LLM gateway: use OpenRouter only for the first version. Do not add the OpenAI SDK or any provider-specific SDK. Implement a small internal Go client around `net/http` for OpenRouter's HTTP API and authenticate only with `OPENROUTER_API_KEY`.
- Storage: use SQLite as the default database for local development and initial production deployments. Use GORM for routine CRUD where it keeps code simple, and use explicit SQL for migrations, FTS5, PRAGMAs, performance-sensitive queries, and any query that GORM makes opaque.
- Cache and queues: use in-process TTL caches plus SQLite-backed durable jobs for v1. Redis is a future scale option, not a default dependency.
- Deployment target: Fly.io. Run the first production version as a containerized Go service on Fly Machines with one primary Machine and one attached Fly Volume mounted at `/data`.

## Goals

1. Answer user questions in Discord with reliable, context-aware LLM responses.
2. Support slash commands, message context menus, buttons, selects, modals, and threaded conversations.
3. Provide admin and moderator tools for guild configuration, permissions, usage controls, and audit logs.
4. Keep user data, secrets, and model interactions observable and governable.
5. Handle rate limits, abuse, large servers, long conversations, file inputs, and partial service outages gracefully.
6. Make the codebase easy to test, operate, and extend with new tools or future LLM gateways.

## Non-Goals For Version 1

- Running arbitrary code generated by the model.
- Training or fine-tuning custom models.
- Replacing Discord's native moderation stack.
- Calling direct provider APIs on day one; all model traffic goes through OpenRouter for now.
- Requiring PostgreSQL, Redis, S3, or an external vector database for the first usable version.
- Running multiple SQLite writers or multi-region replicas in v1.
- Building a public SaaS billing dashboard before the bot works well inside a small set of guilds.

## Product Surface

### User Assistant

- Natural-language messages are the primary user surface.
- Only messages containing the word `Panda` reach the LLM trigger; the trigger then decides whether Panda should respond.
- Natural requests can ask questions, continue chat, summarize, explain, rewrite, translate, or search remembered context without user-facing slash commands.
- Reply support lets users ask follow-up questions against a specific message.
- Thread mode keeps long assistant sessions out of busy channels.
- Ephemeral responses are used for private or sensitive answers.
- Streaming-style updates are simulated with deferred interaction responses and periodic edits.

### Admin And Moderator Tools

- `/admin setup` walks an admin through first-time configuration.
- `/admin model` sets default model, temperature, max response size, and tool policy.
- `/admin prompt` manages server-level system instructions.
- `/admin audit` shows recent privileged actions.
- `/admin disable` pauses assistant responses without kicking the bot.
- Panda chat/tools handle channel rules, role permissions, usage reports, request budgets, guild memory, memory consent, and moderation guidance.
- LLM-triggered Discord components handle reviewed interactive flows such as destructive-action confirmations.

### Owner And Operator Tools

- `/ops health` checks Discord gateway status, SQLite health, queue depth, local storage health, and OpenRouter API status.
- `/ops guilds` lists connected guilds and feature flags.
- `/ops reload` reloads runtime config.
- `/ops drain` gracefully stops handling new jobs on a shard or worker.
- `/ops incident` enables degraded mode banners and stricter limits.
- Owner-only commands are guarded by an allowlist separate from guild admin permissions.

## LLM Capabilities

### OpenRouter Client

- Implement the OpenRouter integration with Go's standard `net/http` client.
- Use `POST https://openrouter.ai/api/v1/chat/completions` for assistant responses.
- Use `Authorization: Bearer $OPENROUTER_API_KEY` for authentication.
- Send `Content-Type: application/json` on all requests.
- Optionally send `HTTP-Referer` and `X-OpenRouter-Title` when configured for app attribution.
- Represent models as OpenRouter model slugs and validate configured slugs through `GET https://openrouter.ai/api/v1/models` where practical.
- Allow `openrouter/auto` as a configuration option when automatic routing is desired.
- Support a primary model plus an optional ordered fallback model list in guild config.
- Support non-streaming first; add streaming via Server-Sent Events after deferred Discord response edits are stable.
- Parse normalized chat completion responses into internal response structs.
- Handle OpenRouter and upstream provider errors distinctly so users see a useful message and operators see actionable telemetry.
- For semantic memory, use `POST https://openrouter.ai/api/v1/embeddings` with an OpenRouter embedding model. If no embedding model is configured, fall back to SQLite FTS5 instead of adding another API key.
- Keep the OpenRouter request and response DTOs inside `internal/llm`.
- Do not import OpenAI SDK packages.

### Core Assistant Behavior

- Use a stable system prompt with server-specific overlays.
- Retrieve Discord context only within configured privacy limits.
- Maintain conversation state per thread or explicit chat session.
- Include citations to Discord messages when summarizing server content.
- Refuse unsafe requests and explain boundaries briefly.
- Ask clarifying questions when the user request is ambiguous.
- Keep answers concise by default and support a `detail` option for longer responses.

### Tool Use

The bot should expose internal tools to the LLM through a controlled tool registry:

- Fetch recent channel or thread messages.
- Fetch a referenced Discord message.
- Search configured knowledge base entries.
- Summarize uploaded text files.
- Create draft moderator notes.
- Read bot configuration available to the current user.
- Generate structured JSON for command workflows.

Every tool must declare:

- Required permission.
- Input schema.
- Output schema.
- Timeout.
- Redaction rules.
- Audit behavior.
- Whether results may be included in model context.

### Memory And Knowledge

- Short-term context lives in an in-process TTL cache with SQLite persistence only when needed for recovery.
- Long-term conversation metadata lives in SQLite.
- Guild knowledge entries are opt-in and admin-managed.
- User-specific memory is off by default and requires clear consent.
- Store embeddings only for explicit knowledge documents or saved summaries, not all Discord traffic.
- Provide export and delete flows for guild and user memory.

## Architecture

### Runtime Components

- Discord gateway worker receives events and interactions.
- Command router validates permissions and dispatches handlers.
- LLM orchestration service builds prompts, runs tools, and calls OpenRouter.
- Context service retrieves Discord messages and conversation state.
- Admin service manages guild config, permissions, limits, and audit logs.
- Moderation service provides review, summarization, and escalation helpers.
- Job worker handles long-running summarization, attachment parsing, and cleanup tasks.
- Fiber HTTP server exposes health checks, metrics, and optional internal admin endpoints.

### Storage Plan

- Primary database: SQLite file at a configurable path, defaulting to `/data/panda.db` on Fly.io and `data/panda.db` locally.
- SQLite driver: start with `gorm.io/driver/sqlite` unless the deployment needs cgo-free builds, then evaluate a pure-Go SQLite driver behind the same store interface.
- ORM: GORM may be used for straightforward CRUD repositories, especially guild config, users, members, conversations, usage events, audit events, and jobs.
- Raw SQL: use explicit SQL for schema migrations, SQLite PRAGMAs, FTS5 virtual tables, triggers, batch operations, query-plan-sensitive search, and maintenance tasks.
- Migrations: use versioned migrations tracked in a `schema_migrations` table. Do not rely on GORM `AutoMigrate` for production schema changes.
- SQLite runtime settings: enable `PRAGMA foreign_keys = ON`, `PRAGMA journal_mode = WAL`, `PRAGMA synchronous = NORMAL`, and a nonzero `busy_timeout` at connection initialization.
- Search: use SQLite FTS5 for local searchable guild knowledge and saved summaries.
- Embeddings: if semantic memory is enabled, generate embeddings through OpenRouter and store vectors in SQLite for small deployments. For larger deployments, add a vector store later behind the memory repository interface.
- Jobs: store durable background jobs in SQLite with status, attempts, lock owner, lease expiration, and last error fields.
- Rate limits: keep fast counters in memory and persist budget windows to SQLite where they must survive restarts.
- Attachment handling: store extracted text and metadata in SQLite. Store temporary downloaded files under a configurable local data directory, defaulting to `/data/tmp`, with TTL cleanup. Do not persist raw attachments by default.
- Backups: provide a command or operator runbook to create SQLite backups, including the main database, WAL, and local attachment cache when enabled.
- Maintenance: periodically run `PRAGMA optimize`; run FTS optimize and `VACUUM` only during maintenance windows or when free pages justify it.

### Fly.io Deployment Plan

- Add `fly.toml` with the app name, primary region, HTTP service config, health check path, and a mount named `data` at `/data`.
- Add a multi-stage Dockerfile that builds a static or minimal Go binary and runs it as a non-root user where practical.
- Bind Fiber to `0.0.0.0:$PORT`; set the Fly internal service port to the same value.
- Create one Fly Volume for v1 and mount it at `/data`. Do not store SQLite data in the image filesystem.
- Run one primary Machine for v1 so SQLite has one writer. Use process-level concurrency inside the app before adding more Machines.
- Store `DISCORD_BOT_TOKEN`, `OPENROUTER_API_KEY`, and other secrets with `fly secrets`; do not bake secrets into `fly.toml`, the image, or `.env`.
- Run migrations on startup before opening Discord event handling. Use a startup lock in SQLite to avoid accidental concurrent migrations if scaling changes later.
- Expose `/healthz` for Fly checks and `/readyz` for dependency readiness.
- Use `fly logs`, `fly status`, `fly releases`, and `fly ssh console` for operations and incident response.
- For multi-machine or multi-region deployments later, choose one of three paths: add LiteFS for SQLite replication, migrate durable state to Fly Postgres, or split stateless workers from the single SQLite-owning bot process.
- For large raw file storage later, evaluate Fly Tigris or another S3-compatible store. Keep it out of v1 unless attachment retention becomes a real requirement.

### Boundary Rules

- `internal/discord` owns Discord SDK types.
- `internal/http` owns Fiber setup, middleware, route registration, and HTTP response shapes.
- `internal/llm` owns OpenRouter request and response DTOs, the HTTP client, and model routing config.
- `internal/store` owns SQLite connection setup, PRAGMAs, migrations, backup, and maintenance.
- `internal/repository` owns GORM-backed repositories and any explicit SQL repositories.
- Command handlers call domain services, not the database directly.
- LLM tools call service interfaces, not Discord or SQL clients directly.
- Public command responses are rendered through a shared response builder to enforce length limits, redaction, and consistent formatting.

## Data Model

Core tables:

- `schema_migrations`: applied migration versions and timestamps.
- `guilds`: guild identity, install status, locale, feature flags.
- `guild_configs`: model defaults, enabled channels, prompt overlays, privacy settings.
- `guild_roles`: Discord role mappings to bot permission names.
- `users`: Discord user identity and global metadata.
- `guild_members`: per-guild user state and consent flags.
- `conversations`: assistant sessions, thread IDs, owners, retention policy.
- `messages`: assistant request and response metadata, not necessarily full message text.
- `knowledge_documents`: admin-managed knowledge source metadata.
- `knowledge_chunks`: indexed chunks and embedding references.
- `knowledge_fts`: SQLite FTS5 virtual table for local keyword search.
- `knowledge_embeddings`: optional OpenRouter embedding vectors for semantic search.
- `attachments`: temporary file metadata, extracted text references, TTL, and cleanup state.
- `usage_events`: token usage, command usage, cost estimates, latency.
- `rate_limit_buckets`: durable budget windows when needed.
- `audit_events`: admin actions, sensitive reads, config changes, memory changes.
- `jobs`: background work state.

Retention rules:

- Store minimal raw Discord content.
- Prefer message IDs, channel IDs, timestamps, hashes, and generated summaries over full transcripts.
- Make raw content retention configurable by guild.
- Delete conversation content after the configured TTL.
- Delete temporary attachment files after extraction or after the configured TTL.
- Keep audit and usage metadata longer than content where legally and operationally appropriate.

SQLite constraints and indexes:

- Use foreign keys for guild-owned and conversation-owned records.
- Add indexes for guild ID, user ID, channel ID, Discord message ID, created time, job status, audit time, and usage time.
- Use composite indexes for common access patterns such as guild plus command, guild plus user, and conversation plus created time.
- Use parameterized queries everywhere. Dynamic column names, sort orders, or table names must come from allowlists.

## Permissions Model

Permission names:

- `assistant.use`
- `assistant.use_threads`
- `assistant.attachments`
- `assistant.memory.read`
- `assistant.memory.write`
- `moderation.use`
- `admin.config.read`
- `admin.config.write`
- `admin.usage.read`
- `admin.audit.read`
- `admin.memory.manage`
- `owner.ops`

Resolution order:

1. Bot owner allowlist.
2. Guild owner.
3. Discord administrator permission.
4. Configured role mappings.
5. Channel-level allow or deny rules.
6. Default guild policy.

All privileged commands should be auditable and should support dry-run output where a change has broad impact.

## Safety, Privacy, And Abuse Controls

- Never log Discord bot tokens, API keys, user access tokens, or raw authorization headers.
- Redact secrets from prompts, logs, traces, and errors.
- Enforce per-guild, per-user, per-channel, and global LLM budgets.
- Add prompt-injection warnings for content fetched from Discord or files.
- Treat fetched Discord content as untrusted model context.
- Require admin opt-in before indexing channel content.
- Use allowlisted tools only; never let the model choose arbitrary network calls.
- Add model output filters for mass mentions, invite spam, credential-looking strings, and unsafe moderator actions.
- Require confirmation buttons for destructive admin operations.
- Keep an audit trail for admin changes, memory operations, and sensitive context reads.

## Observability

Metrics:

- Discord gateway connected shards.
- Interaction latency.
- Command success and failure counts.
- OpenRouter API latency, upstream model latency, and error counts.
- Token usage and estimated cost.
- Queue depth and job age.
- Rate-limit hits.
- SQLite query latency, busy-timeout counts, migration status, and local storage usage.

Logs:

- Structured JSON logs.
- Request IDs per Discord interaction.
- Guild, channel, user, command, and shard identifiers where appropriate.
- Redacted prompt metadata, not raw prompts by default.

Tracing:

- Trace one interaction through command routing, context retrieval, tool calls, LLM call, and Discord response edits.
- Sample traces by default and force tracing for incident mode.

## Testing Strategy

- Unit tests for config validation, permissions, command parsing, context packing, rate limits, and tool schemas.
- Integration tests for SQLite repositories, migrations, PRAGMAs, FTS5 search, and backup behavior.
- Repository tests should use in-memory SQLite where possible and temp-file SQLite when WAL, backup, or filesystem behavior matters.
- Discord adapter tests with mocked interactions and golden response payloads.
- Fiber route tests for health, metrics, and internal admin handlers.
- OpenRouter client tests with fake HTTP servers, recorded fixtures, and schema assertions.
- End-to-end smoke test against a private Discord test guild before releases.
- Load tests for command bursts, summarization jobs, and rate-limit behavior.

## Development Workflow

1. Build vertical slices around user-visible commands.
2. Keep Discord SDK types and OpenRouter DTOs at the edges.
3. Add migrations with every data model change.
4. Add tests before broad refactors.
5. Run formatting, linting, tests, and vulnerability checks before merging.
6. Ship behind guild-level feature flags when behavior is risky.

## Risks And Mitigations

- Discord API changes: isolate Discord library usage and keep command registration idempotent.
- LLM cost spikes: enforce budgets, default limits, admin usage views, and global kill switches.
- Prompt injection: separate trusted instructions from untrusted Discord content and constrain tool use.
- Privacy concerns: make memory opt-in, reduce raw content storage, and provide deletion flows.
- Long-running requests: defer Discord interactions immediately and move heavy work to jobs.
- Library gaps: keep adapter boundaries thin enough to swap DisGo, DiscordGo, or direct REST calls.
- SQLite growth or write contention: use WAL mode, indexes, retention policies, maintenance windows, and a future migration path to PostgreSQL if one SQLite writer becomes a bottleneck.
- Local filesystem loss: document backups and keep raw attachment persistence off by default.
- OpenRouter or upstream model outages: support degraded mode, retries, circuit breakers, fallbacks, and clear user-facing errors.

## First Implementation Slice

1. Initialize the Go module and runtime scaffolding.
2. Add Fiber health route.
3. Initialize SQLite, GORM, migrations, and the storage repositories.
4. Add Dockerfile and `fly.toml` with `/data` volume mount and health check config.
5. Connect to Discord.
6. Register `/ping` and `/admin setup`.
7. Store guild config and usage events.

9. Enforce a basic permission check and per-user rate limit.

This gives the project a working spine before adding memory, summarization, moderation helpers, sharding, and richer admin workflows.

## Assumptions To Confirm

These are assumptions in this plan beyond Go, Fiber, GORM-if-useful, SQLite-if-needed, Discord, and OpenRouter:

- The bot starts as a self-hosted or private-server bot, not a public multi-tenant SaaS.
- Discord gateway events are the primary integration path. Fiber exists for health, metrics, internal APIs, and possible interaction webhooks later.
- One server-side OpenRouter API key pays for model traffic. Users do not bring their own model keys in v1.
- Guild admins configure the bot through Discord slash commands first. A web dashboard is deferred.
- SQLite local disk storage is acceptable for the first production deployment.
- Local filesystem storage is acceptable for temporary attachment processing, with raw attachment persistence disabled by default.
- Redis, PostgreSQL, S3-compatible object storage, and external vector databases are deferred scale options.
- Memory and knowledge indexing are opt-in. The bot does not index entire Discord history by default.
- Moderator features produce suggestions, summaries, and drafts. The bot does not auto-ban, auto-kick, or auto-delete without an explicit later decision.
- Usage budgets are enforced inside the bot rather than through a full billing system.
- The first Fly.io deployment is single-region and starts with one primary Machine attached to one persistent Fly Volume.
- Multi-machine Fly.io scaling is deferred until the project chooses LiteFS, Fly Postgres, or a split stateless-worker architecture.
- Backups are operator-managed at first, using SQLite backup/restore runbooks rather than a managed database service.
- The project favors a clean internal interface over framework lock-in, so Fiber, GORM, DisGo, and OpenRouter stay near their boundaries.

## Source Notes

- [OpenRouter Quickstart](https://openrouter.ai/docs/quickstart) documents direct HTTP API usage against `/api/v1/chat/completions`, which works with any language and avoids SDK lock-in.
- [OpenRouter Authentication](https://openrouter.ai/docs/api/reference/authentication) documents Bearer token authentication with an OpenRouter API key.
- [OpenRouter Chat Completion API](https://openrouter.ai/docs/api/api-reference/chat/send-chat-completion-request) supports streaming and non-streaming chat completion requests.
- [OpenRouter Models API](https://openrouter.ai/docs/api/api-reference/models/get-models) can validate available model slugs and supported parameters.
- [OpenRouter Embeddings API](https://openrouter.ai/docs/api/api-reference/embeddings/create-embeddings) supports embedding requests through the same OpenRouter API key.
- [Fiber documentation](https://docs.gofiber.io/) describes Fiber as an Express-inspired Go framework built on top of Fasthttp.
- [GORM database documentation](https://gorm.io/docs/connecting_to_the_database.html) lists SQLite among the officially supported databases.
- [GORM SQLite driver](https://github.com/go-gorm/sqlite) provides the standard GORM SQLite driver.
- [Fly Volumes overview](https://fly.io/docs/volumes/overview/) explains that Fly Machine root filesystems are ephemeral and persistent state should live on Fly Volumes.
- [Fly SQLite guidance](https://fly.io/docs/rails/advanced-guides/sqlite3/) documents the key deployment constraint for SQLite on Fly: put databases on persistent volumes and start with one node and one volume.
- [Fly LiteFS guide](https://fly.io/docs/litefs/getting-started-fly/) is the later path for distributed SQLite on Fly if multi-machine or multi-region replication becomes necessary.
- [DisGo](https://github.com/disgoorg/disgo) supports Go-based Discord bots with higher-level coverage for gateway, sharding, interactions, application commands, message components, modals, rate limits, webhooks, OAuth2, threads, and voice.
- [DiscordGo](https://github.com/bwmarrin/discordgo) is a mature lower-level Go binding for Discord APIs, including REST endpoints, websocket, and voice support.
- Based on those sources, Go is suitable for this backend without an OpenAI SDK. DisGo is planned as the primary Discord library because its higher-level features better match this bot's interaction-heavy roadmap, with DiscordGo kept as a viable fallback.
